# 专注限额工具 (FocusGuard) 开发过程记录

## 1. 阶段一：基础骨架与监控模块

- **目标**：搭建项目目录结构，完成前台窗口监控。
- **关键实现**：
  - 创建 `focus_guard/` 目录与 6 个核心文件：`main.py`, `config.json`, `monitor.py`, `blocker.py`, `tray_app.py`, `unlock_ui.py`。
  - 初始化 `config.json`，提供 `process_blacklist`（如 `notepad.exe`）和 `title_blacklist`（如 `bilibili`）用于测试。
  - 实现 `monitor.get_active_window_info()`：
    - 使用 Win32 API：`GetForegroundWindow` → `GetWindowText` → `GetWindowThreadProcessId`，再用 `psutil.Process(pid).name()` 获取进程名。
    - 返回值从最初的二元组 `(process_name, window_title)` 优化为三元组 `(process_name, window_title, pid)`，为后续“精准狙击”提供基础。
    - 严格遵守鲁棒性设计：
      - 显式捕获 `psutil.NoSuchProcess` 与 `psutil.AccessDenied`，避免进程瞬间消失或权限不足导致线程崩溃。
      - 顶层 `try/except` 兜底，任何异常都返回 `(None, None, None)`，保持后台线程稳定。
- **测试方式**：
  - 在 `monitor.py` 中加入 `if __name__ == "__main__":` 自测代码，每秒打印当前前台窗口信息，验证进程名与标题获取正确。

## 2. 阶段二：阻断模块 blocker.py

### 2.1 初版设计与问题

- **初版行为**：
  - 简单根据 `process_blacklist` / `title_blacklist` 匹配后，直接按进程名遍历所有同名进程并 `kill()`。
- **暴露问题**：
  - 粒度过粗：当浏览器标题命中 `bilibili` 时，有可能一并杀掉整个浏览器进程，体验过于粗暴。
  - 仅靠线性列表匹配，未来黑名单变长时效率不佳。

### 2.2 优化一：匹配算法与缓存

- **优化点**：
  - 在模块级缓存配置：
    - 将 `process_blacklist` 在初始化时转换为小写 `set`，实现 O(1) 精确匹配。
    - 将 `title_blacklist` 预编译为正则列表 `[re.compile(keyword, re.I), ...]`，避免长列表下的重复字符串搜索。
  - 使用“配置指纹” `_CONFIG_FINGERPRINT`（排序后的元组）判断是否需要重建缓存，避免每次调用都重复构建索引。

### 2.3 优化二：防御性异常与“精准狙击”

- **问题**：Windows 进程状态瞬息万变，拿到 PID 后进程随时可能退出或被系统关闭。
- **改进**：
  - 遍历与 `kill()` 时显式捕获 `psutil.NoSuchProcess` 与 `psutil.AccessDenied`，只打印日志，绝不抛出异常。
  - 引入 PID 级别的 `_kill_process_by_pid(pid, process_name)`：
    - 优先按 PID 杀掉当前前台窗口对应的单个进程。
    - 若失败（无权限或进程已退出），才退回按进程名遍历同名进程。
  - 同时保留 `_kill_processes_by_name`，用于进程黑名单场景的“强力阻断”。

### 2.4 优化三：区分“进程黑名单”与“标题黑名单”

- **需求演进**：
  - 用户希望：黑名单应用（如游戏、桌面 App）可以直接被“秒杀”，但浏览器里只开了一个 Bilibili 标签时，不应把整个浏览器关掉。
- **解决方案**：
  - 将匹配结果拆分为二元组：
    - `matched_process`: 是否命中 `process_blacklist`。
    - `matched_title`: 是否命中 `title_blacklist`。
  - 在 `enforce_rules` 中按类型分支：
    - **命中进程黑名单**：仍使用 PID + 进程名的强力阻断策略（适用于 `notepad.exe` 等本地应用）。
    - **仅命中标题黑名单**：不再调用任何 `kill()`，避免误杀浏览器，仅进行“温和阻断”（见下一小节）。

### 2.5 优化四：对网页的“温和阻断”策略

- **问题**：
  - 对 B 站这类网页，如果只弹通知而不做任何动作，实际效果有限；如果直接杀浏览器进程，又过于激进。
- **最终策略**：
  - 在“仅命中标题黑名单”场景下，采用两步“软拦截”：
    1. 使用 Win32 键盘事件模拟 `Ctrl+W`，尝试关闭**当前活动标签页/文档**，而不是整个进程。
    2. 同时调用 `show_block_warning` 弹出原生系统模态提示，文案优先展示网页标题（例如 `[哔哩哔哩_bilibili]`），以便用户知道是哪个页面被限制。
  - 为防止弹窗风暴，引入全局节流：
    - 字典 `_LAST_WARNING_TIME` 记录每个应用/网页的最近弹窗时间。
    - 限制同一对象在 3 秒冷却时间内只弹一次提示。

### 2.6 优化五：阻断提示的可见性与文案

- **问题过程**：
  - 初版 `MessageBoxW` 只使用 `MB_ICONWARNING | MB_TOPMOST`，在某些情况下不会真正抢到前台，容易被其他窗口遮挡。
  - 调参与测试过程中逐步引入 `MB_SETFOREGROUND` 与 `MB_SYSTEMMODAL`，实现更强的前台展示效果。
  - 初版文案优先显示进程名（如 `msedge.exe`），在网页场景下不友好。
- **最终状态**：
  - 使用组合标志：`MB_ICONWARNING | MB_TOPMOST | MB_SETFOREGROUND | MB_SYSTEMMODAL`，尽可能压过普通窗口（包括资源管理器）。
  - 在网页标题命中的场景下，提示文案优先展示窗口标题，其次才回退到进程名，确保用户看到的是“哪个页面被拦截”，而不是“哪个浏览器进程”。

## 3. 阶段三：托盘与守护进程 (main.py & tray_app.py)

### 3.1 主入口 main.py：状态管理与后台线程

- **状态控制器 `FocusState`**：
  - 通过加锁的布尔变量管理：
    - `is_active`: 是否开启专注模式（默认 `False`）。
    - `stop_flag`: 是否请求停止后台线程，用于“彻底退出”。
  - 提供：
    - `is_active()/set_active()/toggle()` 操作专注状态。
    - `request_stop()/should_stop()` 控制后台监控循环的生命周期。
- **后台监控线程**：
  - `monitor_loop(state, config)` 作为守护线程运行：
    - 每隔 1.5 秒调用 `monitor.get_active_window_info()`。
    - 仅在 `state.is_active()` 为 `True` 时调用 `blocker.enforce_rules()`。
  - 整个循环用 `try/except` 包裹：
    - 任何未预料异常都会被捕获并打印日志，线程不会退出，符合“后台守护线程绝对不能崩”的原则。
- **主线程职责**：
  - 加载配置、创建 `FocusState` 与后台线程。
  - 将 `tray_app.run_tray_app(state)` 作为主线程的阻塞调用，确保符合 pystray 对 UI 线程的要求。

### 3.2 托盘 UI tray_app.py：模式切换与退出

- **图标策略**：
  - 使用 Pillow 在内存中动态生成 64x64 的纯色方块图标：
    - 红色代表“专注模式开启”。
    - 灰色代表“未开启/休息”。
  - 避免对本地图标文件的依赖，减少部署复杂度。
- **菜单设计**：
  - 「切换专注模式」：
    - 调用 `state_controller.toggle()`。
    - 根据当前状态动态切换托盘图标颜色（红/灰），提供即时反馈。
  - 「紧急退出」：
    - 初版仅打印日志，未改变状态，导致用户以为“紧急退出”却依然被拦截。
    - 调整后，在接入解锁 UI 之前的占位实现为：
      - 直接 `state_controller.set_active(False)` 关闭专注模式。
      - 更新托盘图标为灰色，清晰表示当前不再拦截。
    - 为接入 `unlock_ui` 预留扩展位：未来只需在此处插入“子进程解锁成功 → set_active(False)”的逻辑。
  - 「彻底退出」：
    - 调用 `state_controller.request_stop()` 请求后台线程退出。
    - 调用 `icon.stop()` 结束 pystray 事件循环，从而终止主进程。

## 4. 当前状态与后续工作（番茄钟模式前的版本）

- **当前完成度（阶段三结束时）**：
  - 监控模块与阻断模块已按文档第 3.2、3.3 节并结合第 4 节鲁棒性要求实现完毕。
  - 托盘 UI 与守护线程结构稳定，支持：
    - 启停专注模式；
    - 根据专注状态动态调整托盘图标与菜单行为；
    - 完整退出应用。
  - 对应用级黑名单提供“强力秒杀”，对网页级黑名单提供“Ctrl+W 软阻断 + 强提醒”，避免误杀整个浏览器。
- **解锁交互的两轮迭代记录**：
  - **第一轮：两位数乘法 + 承诺语抄写**  
    - 在 `unlock_ui.py` 中先后要求用户完成一道两位数乘法题和一段承诺语抄写，通过退出码 `0/1` 将解锁结果传回主进程。
    - 优点：理论上的阻力极高，基本杜绝“随手解锁”。
    - 问题：乘法难度偏高，很多情况下需要纸笔或计算器，容易带来挫败感，反而促使用户直接彻底退出工具。
  - **第二轮：60 秒强制冷静倒计时**  
    - 将前述解锁流程重构为 60 秒倒计时：
      - 窗口置顶显示“解锁专注模式需要强制冷静。请等待倒计时结束...”，并以大号字体显示“60 秒”倒计时。
      - 在计时未结束前，“确认解锁”按钮为禁用状态，仅允许“放弃解锁，继续专注”或关闭窗口（均视为退出码 1）。
      - 倒计时归零后按钮解锁，点击后以退出码 0 向主进程表明用户坚持完成冷静期。
    - 优点：将“解锁难度”从智力负担转为时间成本，更符合“冷静思考后再决定是否退出”的产品初衷。
    - 问题：解锁本质仍属于“事后惩罚”，同时依旧无法完全阻止用户通过托盘「彻底退出」来绕过设计约束。

## 5. 阶段四（重构）：产品逻辑转向番茄钟严格模式

### 5.1 背景与痛点

- **防作弊漏洞**：
  - 即便有高阻力解锁流程，用户仍然可以通过托盘的「彻底退出」或杀掉主进程来绕过约束，本质上属于“事后毁约”，缺乏充分的心理成本。
  - 在实践中，“紧急退出”+“彻底退出”双通道会让用户在冲动时倾向于直接关掉工具，而不是继续坚持当前专注。
- **体验问题**：
  - 事后惩罚模式依赖复杂的解锁题目和承诺文本，容易造成挫败感或抵触心理。
  - 对真正想专注的用户而言，更自然的心智模型是：**一开始就约定好本轮要专注多久，在这段时间内“彻底焊死退路”**。

### 5.2 决策：转向“事前预承诺”的番茄钟严格模式

- 将产品核心从“事后处罚式解锁”重构为“事前预承诺式番茄钟”：
  - 用户在开始专注前，通过设置界面明确选择本轮专注时长（如 25、45、60 分钟）。
  - 一旦专注开始，在达到结束时间点之前，用户**无法通过 UI 停止专注或退出程序**，只能等待番茄钟自然结束。
  - 通过这种模式，将心理负担前移为“是否愿意承诺这一段时间”，而不是在中途挣扎于“要不要放弃当前承诺”。

### 5.3 技术调整路径

1. **解锁模块重构为时长设置模块**：
   - 原 `unlock_ui.py` 被重构为 `setup_ui.py`：
     - 不再承担“解锁”职责，而是负责在开始专注前收集用户对专注时长的预承诺。
     - 通过退出码传递专注分钟数：`sys.exit(N)` 表示用户承诺专注 N 分钟，`sys.exit(0)` 表示用户取消或关闭窗口，不进入专注状态。
   - 主进程继续通过 `subprocess.run` 阻塞式调用，保持 pystray 与 tkinter 的线程解耦。

2. **主进程引入时间戳管理 (`end_time`)**：
   - 在 `FocusState` 或等效状态管理中新增 `end_time` 字段，用于记录当前专注周期的“到点时间”。
   - 专注开始时：`end_time = now + minutes * 60`。
   - 后台守护线程每轮循环：
     - 若当前时间 `>= end_time`，自动结束专注，将 `is_active` 设为 False，并重置 `end_time`。
     - 若当前时间 `< end_time`，则继续执行阻断逻辑，忽略一切试图“提前解锁”的请求。

3. **托盘菜单与退出路径的策略性收紧**：
   - 托盘菜单改造为：
     - 【开始专注】：仅在非专注状态下可用，点击后调用 `setup_ui.py` 让用户选择时长。
     - 【专注中（不可退出）】：用于展示当前处于专注期，控件本身不可点击。
     - 【彻底退出】：在非专注状态下可用；一旦专注开始，整个专注周期内完全禁用（置灰），防止通过 UI 直接关闭进程。
   - 配合上述策略，“专注期间彻底焊死退路”：除非用户从系统级别强制结束进程，否则无法通过正常交互提前终止专注，有效提升承诺的约束力。

### 5.4 边界漏洞与体验优化（编码前设计）

在实现番茄钟严格模式前，通过设计审查发现两个关键边界漏洞，需在编码阶段提前规避：

- **缺陷一：sys.exit 陷阱（崩溃退出码陷阱）**
  - **问题描述**：`setup_ui.py` 作为独立子进程，若发生未捕获异常崩溃，默认会以退出码 1 结束。若主进程直接接收 `returncode == 1` 并视为“用户承诺 1 分钟专注”，会导致用户被意外锁死 1 分钟，体验极差。
  - **决策**：主进程增加返回码有效性校验底线，**仅当 `returncode >= 5` 时才视为有效专注时长**（设定最小专注时长底线为 5 分钟）。若返回码 `< 5` 或 `== 0`，一律视为无效或取消，不进入专注状态。这样即使子进程崩溃返回 1，也不会误判为用户承诺。

- **缺陷二：刑满释放无感知（番茄钟结束无提示）**
  - **问题描述**：原设计中，番茄钟到点后仅在后台默默将 `is_focus_mode_active` 设为 False，用户无法及时得知可以休息，容易造成“专注已结束但仍在自我约束”的困惑。
  - **决策**：在后台循环检测到 `当前时间 >= end_time` 并自动结束专注时，**必须调用 `ctypes.windll.user32.MessageBoxW` 弹出系统原生提示框**，标题为“FocusGuard 专注完成”，内容为“专注番茄钟已完成！辛苦了，休息一下吧”，确保用户及时感知到可以休息，提升完成专注的仪式感和反馈感。

## 6. 阶段五：前端架构现代化、可视化看板与缓解时间焦虑

### 6.1 背景与痛点

- **痛点一：手动修改 JSON 的极客门槛**
  - 用户需要手动编辑 `config.json` 来管理黑名单，极易出现 JSON 格式错误（如缺少逗号、引号不匹配），导致配置文件损坏、程序无法启动。
  - 对于非技术用户，查找进程名、判断哪些应用需要加入黑名单，本身就是高门槛操作。
  - 缺乏“所见即所得”的可视化界面，无法直观看到当前黑名单状态，也难以快速添加正在运行的进程。

- **痛点二：严格模式下的“盲专注”时间焦虑**
  - 番茄钟严格模式虽然有效提升了约束力，但用户在整个专注期间无法看到剩余时间，容易产生“已经专注了多久？还有多久结束？”的焦虑感。
  - 缺乏视觉反馈，用户可能频繁查看系统时间，反而分散注意力。
  - 原设计中，即使有完成提示弹窗，但在专注进行中用户完全处于“盲状态”，体验不够友好。

### 6.2 决策：前端架构现代化与可视化看板

- **UI 库升级：从 tkinter 到 CustomTkinter**
  - 将现有所有 GUI 模块（`setup_ui.py`、新增的 `dashboard_ui.py`、`timer_widget.py`）统一迁移至 **CustomTkinter**。
  - 优势：
    - 原生支持深色模式，符合现代应用审美。
    - 支持无边框窗口（`overrideredirect`）与圆角设计，适合悬浮小部件。
    - API 与 tkinter 高度兼容，迁移成本低。
    - 提供更丰富的现代化组件（圆角按钮、卡片式布局等）。

- **主控看板模块 (`dashboard_ui.py`)：可视化黑名单管理 + Smart Picker**
  - **功能 A：可视化黑名单管理**
    - 提供列表/表格界面，实时展示 `process_blacklist` 与 `title_blacklist`。
    - 支持增删改查，所有操作实时同步到 `config.json`，无需手动编辑。
    - 通过托盘菜单【打开主控看板】以非阻塞子进程方式启动，仅在非专注状态下可用。
  - **功能 B：活跃进程嗅探器 (Smart Picker)**
    - 调用 `psutil` 列出当前有窗口界面的活跃进程，展示进程名、PID、窗口标题。
    - 每个进程项提供“一键加入黑名单”按钮，点击后自动添加到 `process_blacklist` 并写入配置。
    - 提供“刷新进程列表”功能，定期更新快照。
    - 彻底解决“如何知道进程名”、“如何快速添加正在运行的应用”的用户痛点。

- **悬浮倒计时模块 (`timer_widget.py`)：缓解时间焦虑**
  - 在专注开始时，通过 `subprocess.Popen` 非阻塞启动独立的倒计时子进程。
  - 子进程接收 `end_time` 作为命令行参数，每秒计算剩余时间并以“MM:SS”格式显示。
  - 窗口特性：
    - 无标题栏（`overrideredirect(True)`）、置顶、不可点击（仅作展示）。
    - 极简设计，固定在屏幕右上角或用户自定义位置。
    - 当剩余时间 `< 60` 秒时，切换为红色高亮，增强紧迫感。
  - 倒计时归零时，子进程自动 `sys.exit(0)` 销毁，不干扰主进程的完成提示。
  - 通过这种“非阻塞子进程 + 独立窗口”的设计，既提供了视觉反馈，又完全不影响底层守护线程的稳定性与鲁棒性。

### 6.3 技术实现要点

- **进程通信与解耦**：
  - `dashboard_ui.py` 与 `timer_widget.py` 均以独立子进程运行，通过文件系统（`config.json`）或命令行参数（`end_time`）与主进程通信。
  - 主进程不直接调用 CustomTkinter，保持 pystray 主线程的纯净性，避免 UI 线程死锁。

- **配置同步机制**：
  - `dashboard_ui.py` 修改 `config.json` 后，主进程的 `blocker.py` 模块在下次调用 `_ensure_rule_index` 时会通过配置指纹检测变化，自动重建缓存。
  - 无需重启主进程，配置变更即时生效。

- **专注标签预留**：
  - 在 `setup_ui.py` 中新增“专注目标/标签”输入框，标签信息暂存本地日志，为未来数据统计（如“本周专注了哪些目标”、“各目标累计时长”）做预留。

## 7. 阶段五补充：系统弹性、UI 优化与高阻力安全阀

### 7.1 背景与痛点

- **痛点一：悬浮窗固定位置遮挡工作区**
  - 悬浮倒计时窗口固定在屏幕右上角，在某些工作场景下（如全屏代码编辑器、多显示器布局）可能遮挡重要内容。
  - 用户无法根据当前工作环境灵活调整窗口位置，影响使用体验。

- **痛点二：绝对焊死退路缺乏弹性**
  - 严格模式下，专注期间“彻底退出”菜单完全禁用，虽然有效提升了约束力，但在遇到真实紧急情况（如工作突发需求、紧急会议）时缺乏可控的退出路径。
  - 用户可能因无法通过正常交互退出而产生“暴力杀进程”的冲动，反而破坏了工具的权威性和用户信任。

### 7.2 决策：系统弹性与高阻力安全阀

- **赋予悬浮窗拖拽与隐藏能力**：
  - 在 `timer_widget.py` 中实现鼠标拖拽功能（绑定 `<Button-1>` 与 `<B1-Motion>` 事件），允许用户将悬浮窗拖动到屏幕任意位置，避免遮挡工作区。
  - 提供隐藏按钮（如右上角 "×"），点击后仅销毁当前 UI 进程，不影响主进程的专注状态。
  - 主进程新增 `show_timer_widget()` 方法，支持通过托盘菜单【显示倒计时】随时重新拉起悬浮窗，确保用户不会因误操作而永久失去视觉反馈。

- **引入紧急安全阀 (`emergency_ui.py`)**：
  - 作为专注期间唯一的可控退出路径，通过“60 秒强制冷静期”的极高时间成本，既保证了真实突发情况的出口，又有效遏制了多巴胺驱使下的随手毁约行为。
  - 与“彻底退出”菜单在专注期间被禁用形成对比，紧急安全阀是专注期间唯一可控的退出路径，但需要付出足够的时间成本。
  - 主进程新增 `emergency_stop()` 方法，专门处理紧急退出逻辑，确保状态清理与 UI 恢复的一致性。

### 7.3 技术实现要点

- **拖拽交互实现**：
  - 记录鼠标按下时的窗口位置 `(x, y)` 与鼠标位置 `(mouse_x, mouse_y)` 的差值。
  - 在 `<B1-Motion>` 事件中，根据当前鼠标位置与初始差值的偏移量，动态更新窗口位置 `root.geometry(f"{width}x{height}+{new_x}+{new_y}")`。

- **进程生命周期管理**：
  - `FocusState` 类维护 `timer_process` 对象，在 `start_focus()` 时启动，在 `set_active(False)` 或 `emergency_stop()` 时清理。
  - `show_timer_widget()` 方法检查进程状态，若已退出则重新启动，避免重复创建进程。

- **安全阀的阻塞式调用**：
  - 紧急安全阀通过 `subprocess.run()` 阻塞调用，确保用户必须完成整个冷静期流程才能退出，中途放弃则返回码为 1，主进程保持专注状态不变。

## 8. 阶段六：从后台脚本到现代化桌面应用

### 8.1 背景与痛点

- **痛点一：每次需通过命令行启动，且依赖托盘右键交互，操作路径长且原始**
  - 用户需要手动打开命令行，运行 `python main.py`，然后通过托盘右键菜单进行所有操作。
  - 缺乏标准的桌面应用启动方式（双击图标、自动启动主窗口），不符合现代软件的用户预期。
  - 托盘右键菜单虽然功能完整，但交互路径较长，不够直观。

- **痛点二：缺乏直观的规则配置界面，手动修改 JSON 体验极差**
  - 虽然已有 `dashboard_ui.py` 规划，但尚未实现，用户仍需手动编辑 `config.json`。
  - 缺乏“所见即所得”的可视化界面，无法直观看到当前黑名单状态，也难以快速添加正在运行的进程。
  - 对于非技术用户，查找进程名、判断哪些应用需要加入黑名单，本身就是高门槛操作。

- **痛点三：缺乏专注数据的可视化反馈，难以形成多巴胺正向激励**
  - 虽然 `focus_log.txt` 已记录专注历史，但用户无法直观看到“今天专注了多久”、“累计专注了多久”等统计数据。
  - 缺乏数据可视化反馈，难以形成“完成专注 → 看到数据增长 → 获得成就感 → 继续专注”的正向循环。

### 8.2 决策：确立控制中心为全局图形化入口

- **重定位 `dashboard_ui.py` 为 FocusGuard 控制中心**
  - 将 `dashboard_ui.py` 从“可选的黑名单管理工具”升级为“程序的主控窗口”，作为全局图形化入口。
  - 采用 CustomTkinter 的 **TabView** 组件（或侧边栏导航）进行功能路由，打造 All-in-One 的现代软件体验。
  - 三大核心功能区：
    1. **专注台 (Focus)**：接管原 `setup_ui.py` 的功能，集成专注目标输入、时长选择与启动逻辑。
    2. **规则库 (Blacklist)**：实现可视化黑名单管理与 Smart Picker 进程一键抓取功能，UI 操作直接修改 `config.json`。
    3. **统计局 (Statistics)**：读取并解析 `focus_log.txt`，展示“今日总时长”、“累计总时长”以及历史专注记录列表。

- **启动与交互流现代化**
  - `main.py` 启动时，除了挂载托盘图标，应默认通过 `subprocess.Popen` 非阻塞拉起 `dashboard_ui.py` 主窗口，作为程序的图形化入口。
  - 确立“双击托盘图标呼出主界面”的标准桌面软件交互范式：托盘图标的双击事件或左键点击事件，应绑定为“显示控制中心”，检查子进程是否仍在运行，若已退出则重新启动。
  - 托盘右键菜单中的【打开主控看板】项改为【显示控制中心】，功能与双击事件一致。

- **废弃与整合**
  - `setup_ui.py` 将被完全废弃，其所有功能（专注目标输入、时长选择、日志记录）均合并入 `dashboard_ui.py` 的“专注台”标签页中。
  - 托盘菜单中的【开始专注】项将被移除，用户统一通过控制中心的“专注台”启动专注。
  - 通过这种整合，将“极客后台脚本”彻底进化为“现代化桌面应用”，提供统一的图形化入口和直观的操作界面。

### 8.3 技术实现要点

- **进程间通信**：
  - `dashboard_ui.py` 的“专注台”标签页中，用户点击【开始专注】后，将专注信息（目标、分钟数）写入临时文件 `focus_command.json`。
  - 主进程的监控循环定期检查该文件，读取后删除文件并调用 `state.start_focus(minutes)`。
  - 通过文件系统通信，避免复杂的命名管道或 socket 实现，保持架构简洁。

- **数据统计实现**：
  - 使用 `Path(__file__).with_name("focus_log.txt")` 定位日志文件。
  - 逐行解析，格式为：`YYYY-MM-DD HH:MM:SS | 目标文本 | 分钟数`。
  - 计算今日总时长：筛选今日日期（`datetime.now().date()`）的记录，累加时长。
  - 计算累计总时长：累加所有记录的时长。
  - 若日志文件不存在或格式异常，优雅降级：显示“暂无数据”或“0 小时 0 分钟”。

- **窗口生命周期管理**：
  - `main.py` 维护 `dashboard_process` 对象，跟踪控制中心子进程状态。
  - 托盘双击/左键点击时，检查进程是否仍在运行，若已退出则重新启动。
  - 控制中心窗口关闭时，可选择仅隐藏窗口（不退出子进程）或询问用户是否退出程序。

### 8.4 阶段六第一步测试后的两处逻辑修复

在第一步（打通主控看板与通信链路）的测试中，发现两个严重逻辑漏洞，已按下列方式修补并写入本文档。

**漏洞一：专注期间仍可呼出控制中心（防作弊漏洞）**

- **问题**：专注由控制中心写 JSON 异步触发后，用户仍可通过托盘菜单或左键点击托盘图标再次打开控制中心，存在作弊或误操作空间。
- **决策**：专注期间彻底禁用“显示控制中心”入口，且不强制关闭已运行中的 dashboard 进程（点击时长按钮时 dashboard_ui 已自行 `sys.exit(0)`），仅确保无法被重新唤起。
- **实现**：
  - 托盘菜单【显示控制中心】项增加 `enabled=lambda item: not state_controller.is_active()`，仅非专注状态下可用。
  - 该菜单项设为 `default=True`，作为托盘图标左键单击的默认动作；专注时项被禁用，且回调内第一行增加判断：`if state_controller.is_active(): return`，专注期间即使被触发也不执行打开逻辑。

**漏洞二：托盘图标变色失效（异步触发导致更新时机丢失）**

- **问题**：专注由 `monitor_loop` 读取 `focus_command.json` 异步触发，托盘图标未在“专注开始”时变红；专注时间到期后由 `monitor_loop` 自动调用 `set_active(False)`，图标也未变灰。
- **决策**：将“更新托盘图标”封装为可由状态变化处统一触发的回调，在 `start_focus`、`set_active(False)`、`emergency_stop()` 完成后立即调用，保证图标与状态一致。
- **实现**：
  - 在 `FocusState` 中新增可选回调 `_on_icon_update` 与 `set_icon_update_callback(callback)`，由 `tray_app` 在创建托盘图标后注册 `lambda: update_icon(icon)`。
  - 在 `set_active()`、`start_focus()`、`emergency_stop()` 的逻辑末尾（在释放锁之后）调用 `_notify_icon_update()`，从而在 `monitor_loop` 异步执行 `state.start_focus(minutes)` 或 `state.set_active(False)` 时也能立刻同步更新托盘图标。
  - 回调在未持锁状态下调用，且用 try/except 包裹，避免死锁或异常导致监控线程崩溃。
